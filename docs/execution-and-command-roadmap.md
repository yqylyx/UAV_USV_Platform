# 任务执行与控制指令开发路线

## 模块边界

- `simulation_session` 只管理 ROS/Gazebo 与 Unity 仿真环境的启动、停止和接入状态。
- `mission_task` 保存可重复使用的任务配置。
- `mission_run` 保存任务每一次实际执行，暂停、恢复和结束均作用于同一执行批次。
- `control_command` 保存单条指令从创建到外部组件确认的完整生命周期。
- `device_runtime_status` 保存最新状态，`device_telemetry` 保存可回放的历史遥测。

## 当前完成状态

数据库 V9 已完成：

- 新增 `mission_run`。
- 控制指令增加任务批次、目标设备、唯一指令编号、载荷、下发时间、确认时间和错误码。
- 遥测增加任务批次和仿真会话归属。
- 任务事件增加任务批次、阶段和事件等级。

后端已完成：

- 同一任务可生成多次执行批次，历史相互独立。
- 运行中任务禁止编辑和删除。
- 控制指令状态支持 `PENDING`、`DISPATCHED`、`ACKNOWLEDGED`、`FAILED`、`TIMEOUT`。
- 提供外部组件指令确认入口。
- 本地默认使用明确标识的 Mock 指令适配器。
- 任务开始、暂停、恢复、完成、异常和取消均由控制指令确认事件驱动状态变更。
- 指令失败或超时时不会把任务错误地推进到目标状态，并会写入任务告警事件。
- 同一任务批次只允许存在一条等待确认的控制指令。

## 后续顺序

1. 实现 ROS 指令适配器，逐步替换 Mock 适配器。
2. 建立通信链路当前状态、历史采样和告警记录。
3. 基于 `mission_run + device_telemetry + mission_event` 实现回放。
4. 增加任务评估指标和报告导出。

## 开发约束

- 已执行的 Flyway 迁移禁止修改，后续数据库变更从 V10 开始。
- Unity WebGL 构建产物不在前后端业务迭代中修改。
- 指令“已记录”“已下发”“已确认”必须分别展示。
- 新增按钮前必须明确对应接口、权限、数据库记录和失败反馈。
